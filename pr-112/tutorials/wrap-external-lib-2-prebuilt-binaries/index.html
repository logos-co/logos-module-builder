    Tutorial 2 — Prebuilt binaries in the same repo¶

    Goal: wrap a library you only have as prebuilt binaries (.so/.dylib/ .a), committed in your module repo — and ship builds for several platforms at once.

    When to use: the library is proprietary, slow to build, or produced by a separate toolchain (cross-compiled, vendor-supplied, CI artifact). You commit the binaries and the build links + bundles them.

    Runnable doctest

    The config/code blocks below are transcluded from the worked example at tests/fixtures/extlib-doctests/case2-binaries, which ships prebuilt libgreet for all four platforms and is built + checked in CI (checks.<system>.extlib-case2-binaries).

    The platform-disambiguation problem¶

    A single-platform vendored library is easy — drop it in lib/ and point vendor_path at it. But the moment you support more than one platform you hit a naming collision: linux x86_64 and linux aarch64 both use libgreet.so. They can't both sit in lib/.

    The solution: per-platform subdirectories¶

    Put each platform's binary in a subdirectory named by its Nix system string. The build automatically selects the subdirectory matching the system it is building for; the shared header stays flat:

    greetbin/
├── flake.nix
├── metadata.json
├── CMakeLists.txt
├── lib/
│   ├── greet.h                       # shared header, flat
│   ├── x86_64-linux/libgreet.so
│   ├── aarch64-linux/libgreet.so
│   ├── x86_64-darwin/libgreet.dylib
│   └── aarch64-darwin/libgreet.dylib
└── src/
    ├── greetbin_interface.h
    ├── greetbin_plugin.h
    └── greetbin_plugin.cpp

    The four valid subdirectory names are exactly the Nix systems Logos targets:

    Subdirectory Platform
    x86_64-linux Linux, Intel/AMD 64-bit
    aarch64-linux Linux, ARM 64-bit
    x86_64-darwin macOS, Intel
    aarch64-darwin macOS, Apple Silicon

    These are Nix system strings

    They are what nix eval --raw --expr builtins.currentSystem prints — not the .lgx packaging variant labels (linux-amd64, darwin-arm64). Don't confuse the two.

    You only need to commit the platforms you actually support. A flat lib/libgreet.<ext> (no subdir) is the single-platform form and keeps working unchanged — just don't mix the two layouts for the same library.

    metadata.json¶

    The external_libraries entry is the same as for a single-platform vendored lib — vendor_path points at the directory holding the per-platform subdirs:

    metadata.json
    {
  "name": "greetbin",
  "version": "1.0.0",
  "type": "core",
  "category": "integration",
  "description": "Wraps a prebuilt libgreet binary vendored per-platform under lib/<nix-system>/",
  "main": "greetbin_plugin",
  "dependencies": [],

  "nix": {
    "packages": {
      "build": [],
      "runtime": []
    },
    "external_libraries": [
      { "name": "greet", "vendor_path": "lib" }
    ],
    "cmake": {
      "find_packages": [],
      "extra_sources": [],
      "extra_include_dirs": ["lib"],
      "extra_link_libraries": []
    }
  }
}

    flake.nix¶

    No externalLibInputs — the binaries are in the repo:

    flake.nix
    {
  description = "greetbin — Logos module wrapping a per-platform prebuilt libgreet binary";

  inputs = {
    logos-module-builder.url = "github:logos-co/logos-module-builder";
  };

  outputs = inputs@{ logos-module-builder, ... }:
    logos-module-builder.lib.mkLogosModule {
      src = ./.;
      configFile = ./metadata.json;
      flakeInputs = inputs;
      # No externalLibInputs: the library is a prebuilt binary committed under
      # lib/<nix-system>/ and selected automatically by the build.
    };
}

    CMakeLists.txt¶

    EXTERNAL_LIBS greet finds, links, adds lib/ as an include dir, and copies the shared library next to the plugin:

    CMakeLists.txt
    # greetbin — wraps a prebuilt libgreet binary vendored per-platform.
cmake_minimum_required(VERSION 3.14)
project(GreetbinModulePlugin LANGUAGES CXX)

# Include the Logos Module CMake helper (provided via LOGOS_MODULE_BUILDER_ROOT
# during a nix build).
if(DEFINED ENV{LOGOS_MODULE_BUILDER_ROOT})
    include($ENV{LOGOS_MODULE_BUILDER_ROOT}/cmake/LogosModule.cmake)
elseif(EXISTS "${CMAKE_CURRENT_SOURCE_DIR}/cmake/LogosModule.cmake")
    include(cmake/LogosModule.cmake)
else()
    message(FATAL_ERROR "LogosModule.cmake not found")
endif()

# Derive module identity from metadata.json
file(READ "${CMAKE_CURRENT_SOURCE_DIR}/metadata.json" METADATA_JSON)
string(JSON MODULE_NAME GET ${METADATA_JSON} name)
string(JSON MODULE_VERSION GET ${METADATA_JSON} version)
file(WRITE "${CMAKE_CURRENT_BINARY_DIR}/module_config.h"
    "#pragma once\n"
    "#define MODULE_NAME \"${MODULE_NAME}\"\n"
    "#define MODULE_VERSION \"${MODULE_VERSION}\"\n"
)
include_directories("${CMAKE_CURRENT_BINARY_DIR}")

# EXTERNAL_LIBS greet => find_library(libgreet) in lib/, link it, add lib/ as an
# include dir, and copy the shared lib next to the plugin. The builder stages the
# platform-matching binary from lib/<nix-system>/ into lib/ before this runs.
logos_module(
    NAME ${MODULE_NAME}
    SOURCES
        src/greetbin_interface.h
        src/greetbin_plugin.h
        src/greetbin_plugin.cpp
    EXTERNAL_LIBS
        greet
)

    Plugin code¶

    hello() calls greet_hello() from the vendored binary via the flat header:

    lib/greet.h
    #ifndef GREET_H
#define GREET_H

#ifdef __cplusplus
extern "C" {
#endif

/* Returns a static greeting string owned by the library. */
const char* greet_hello(void);

#ifdef __cplusplus
}
#endif

#endif /* GREET_H */
    src/greetbin_plugin.cpp
    #include "greetbin_plugin.h"
#include "logos_api.h"
#include "lib/greet.h"
#include <QDebug>

GreetbinPlugin::GreetbinPlugin(QObject* parent)
    : QObject(parent)
{
}

GreetbinPlugin::~GreetbinPlugin() = default;

void GreetbinPlugin::initLogos(LogosAPI* api)
{
    // Assign the inherited PluginInterface::logosAPI member — the host reads it
    // directly to dispatch cross-module calls.
    logosAPI = api;
}

QString GreetbinPlugin::hello()
{
    // Calls into the vendored libgreet binary. If the wrong-arch binary were
    // selected (or none), this symbol would be unresolved at load time.
    const QString out = QString::fromUtf8(greet_hello());
    emit eventResponse("hello", QVariantList() << out);
    return out;
}

    (The full greetbin_interface.h / greetbin_plugin.h are in the fixture.)

    How selection works¶

    git add your binaries (Nix only sees git-tracked files), then build:

    ws build greetbin                                  # or: nix build
lm methods ./result/lib/greetbin_plugin.so
logoscore -m ./result/lib -l greetbin -c "greetbin.hello()"
# -> "hello from libgreet"

    During the build, mkExternalLib detects the lib/<nix-system>/ subdir for the build platform and stages that one binary into lib/, where CMake's find_library picks it up. The bundled libgreet.<ext> is placed next to the plugin and the RPATH (@loader_path on macOS, $ORIGIN on Linux) resolves it at runtime. On macOS, install_name/@rpath are fixed up automatically.

    Producing the binaries¶

    Build libgreet for each platform with your normal toolchain (cc -shared -fPIC -o libgreet.so greet.c), or cross-compile all four with zig (handy for ELF + Mach-O from one host):

    zig cc -target x86_64-linux-gnu  -shared -fPIC -Wl,-soname,libgreet.so      -o lib/x86_64-linux/libgreet.so      greet.c
zig cc -target aarch64-linux-gnu -shared -fPIC -Wl,-soname,libgreet.so      -o lib/aarch64-linux/libgreet.so     greet.c
zig cc -target x86_64-macos  -shared -fPIC -Wl,-install_name,@rpath/libgreet.dylib -o lib/x86_64-darwin/libgreet.dylib  greet.c
zig cc -target aarch64-macos -shared -fPIC -Wl,-install_name,@rpath/libgreet.dylib -o lib/aarch64-darwin/libgreet.dylib greet.c

    Give shared libraries a SONAME (Linux) / install_name (macOS) of the bare filename (libgreet.so / @rpath/libgreet.dylib) so the plugin records a relocatable dependency that the RPATH can resolve. A vendored .a static archive is linked in at build time and needs neither.

    Notes & limitations¶

    • The doctest verifies the correct architecture was selected (file on the staged binary) and that the plugin genuinely links it (otool -L / readelf -d showing the dependency) — important on macOS, where -undefined dynamic_lookup would otherwise let a missing library link silently.
    • Raw nix develop + cmake does not descend into lib/<nix-system>/; the subdir selection happens during the Nix build's staging step. Build with ws build / nix build (the supported path). For raw-cmake iteration, copy the right subdir's binary into lib/ yourself.
